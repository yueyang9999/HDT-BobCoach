# HDT-BobCoach 公开产品仓库治理设计

日期：2026-07-20
状态：已批准（方案 A）

## 决策

建立全新的公开仓库 `yueyang9999/HDT-BobCoach`，只承载可分发的 HDT 插件产品、自动化测试、确定性发布工具和公开文档。旧仓库 `yueyang9999/HDT-BObcoash` 转为私有并归档，只作为只读历史来源。

不迁移旧 Git 历史。旧历史已经公开包含大量 replay XML、原始日志、个人路径和无关工程内容；在新仓库重放历史会延续暴露面并让清理失去意义。

## 产品边界

生产代码以 `BobCoach.csproj` 的显式依赖为准，迁移到 `src/BobCoach/`。运行所需本地规则数据由项目文件显式嵌入或由发布构建明确复制。

测试迁移到 `tests/`，只保留能验证生产边界、构建身份、离线包、安装生命周期和核心行为的代码。Node.js 仅作为无第三方运行时依赖的合同测试驱动，不再承载 Electron 产品路线。

构建和发布工具分别放入 `tools/build/` 与 `tools/release/`。生成的 DLL、ZIP、校验和与验收证据不提交到 Git。

## 公开合规

- 不含 replay XML、用户日志、账户信息、私人绝对路径、密钥或本机配置。
- 不含 `.debate`、`.mcp`、`sessions`、`local-data`、VM、ISO、截图或 P0 证据。
- 不含 Python 分析管线、Electron overlay、`BobObserver` 或无关实验。
- 保留 MIT 许可证、第三方声明、数据来源、隐私、安全、支持和赞赏边界。
- 赞赏只通过仓库外页面提供，完全自愿，不解锁功能，不出现在 Hearthstone 或 HDT overlay 内。

## 发布边界

仓库 push、CI 和 VM overlay 验收属于本次批准范围。GitHub Release 是面向用户的正式公开发布，必须在最终包与 VM 验收通过后再次取得明确授权。

## 验收标准

1. 新仓库可从干净 checkout 构建和测试。
2. 发布脚本产生严格文件白名单的离线 ZIP，并输出 SHA-256。
3. 仓库扫描不含敏感文件、replay 数据、私人路径或异常大文件。
4. GitHub 默认分支、社区健康文件、Issue/PR 模板与 CI 正确。
5. 旧仓库为 `PRIVATE + archived`。
6. Win11 24H2 临时 overlay 中，以标准用户、无 Node、无管理员权限、离线完成安装验收；Win10 作为兼容性补充。
